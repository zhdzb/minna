# Backend Architecture & Prompt Design Plan

The goal is to design a backend logic capable of providing context-aware, leveled Japanese exercises using the Gemini API, while minimizing API calls and handling the lack of persistent conversational state in the API itself.

## 核心挑战与解决思路

Gemini API (REST/SDK) 本身是无状态的（Stateless），它不知道您“目前学到了第几课”或者“上一道题做错了什么”。因此，我们需要在本地（我们的 Node.js/Browser JS 逻辑中）构建一个**坚固的上下文管理器（Context Manager）**，每次向 Gemini 发请求时，都把当前的必要规则、背景信息一并带上。

同时，为了提高效率、减少 API 调用次数（省额度、加快响应），正如您建议的，我们采用**“按类批量生成 (Batch Generation by Category)”**的策略，而不是做一题请求一次。

## 二、 核心 AI 交互流 (API 调用策略)

我们将与 Gemini 的交互分为两个主要接口（逻辑上的，实际都在前端/Node调取）。

### 1. 批量出题引擎 (Batch Exercise Generator)

**触发时机**：当用户进入某一课（例如第 6 课），或者点击“开始一组练习”时。一次 API 调用，返回多道不同类型的题目。

**Prompt 设计范式 (System + User Prompt)**:

```text
[System Instruction]
你是一个专业的日语教师，精通《大家的日语》教材。现在的任务是为学习者生成一套进阶练习题。
遵循准则：
1. 词汇与语法限制：绝对不能超纲！只能使用《大家的日语》第 1 课 到 第 {当前课数} 课中出现过的单词和语法。
2. 题型多样性：请生成 3 道【Level 1: 基础助词填空】，3 道【Level 2: 降维造句】，1 道【Level 3: 职场情景对话阅读】。
3. 格式要求：必须严格返回纯 JSON 格式，不要包含任何 markdown 标记 (如 ```json)。

[User Input (Dynamic Inject)]
当前学习进度：第 6 课。
请根据第 6 课的重点句型（如 を、で、に动词接续）生成练习。
如果有以下错题历史，请尝试将类似的语法点融入新的题目中进行复盘测试：
[错题历史点：第5课的 方向助词 へ，第4课的时间词不加助词]
```

**期望的 JSON 返回结构 (Gemini -> App)**:
*通过强制要求返回 JSON，我们的应用可以直接解析并渲染到界面上，无需每次动态请求。*
```json
{
  "exercises": [
    {
      "id": "ex_6_001",
      "type": "level_1_fill",
      "question": "わたしは スーパー (   ) 肉を 買います。",
      "options": ["で", "に", "へ", "を"],
      "answer": "で",
      "explanation": "表示动作发生的场所，用助词『で』。"

    {
      "id": "ex_6_002",
      "type": "level_2_translate",
      "chinese_prompt": "我每天早上吃鸡蛋。",
      "vocab_hints": [
        {"word": "毎日", "kana": "まいにち", "cn": "每天"},
        {"word": "朝", "kana": "あさ", "cn": "早上"},
        {"word": "卵", "kana": "たまご", "cn": "鸡蛋"},

      "answer_pattern": "わたしは まいにち あさ たまごを たべます。"

    // ... 其他题目

### 2. 智能批改与讲解引擎 (Smart Evaluator)

**触发时机**：用户做好了**一批**造句题（Level 2），点击“提交批改”。（填空题前端直接对答案，无需消耗 API）。我们将这批造句题一起发给 Gemini 批改，再次节省请求。

**Prompt 设计范式**:

```text
[System Instruction]
你负责批改日语初学者的造句练习。
对于每一个用户提交的答案，你需要：
1. 判定对错 (is_correct: boolean)。允许细微的假名/汉字转换差异。
2. 如果错误，指出具体错误类型（如：助词错误、动词变形错误、时态错误）。
3. 给出标准答案（使用《大家的日语》第 {当前课} 课及以前的语法）。
4. 给出简短的中文讲解，说明为什么错。
5. 给出1句更地道、适用于赴日工作职场/生活的扩展表达 (natural_expression)。
返回严格的 JSON 数组结构。

[User Input]
用户当前阶段：第 6 课。
批改任务列表：
1. 题目："我用日语写信。" 用户答案："watashi wa nihongo ni tegami o kakimasu" (注意：用户提交的可能是罗马音，请将其视作对应的假名进行批改，并在解析中附上正确的假名写法)
2. 题目："一起喝啤酒吧？" 用户答案："いっしょに ビールを 飲みませんか。"
```

## 三、 遗漏的关键点与进阶设计建议 (Romaji & Edge Cases)

您提到的“只能用罗马音输入”是一个非常典型且极其重要的初学者痛点！针对这点以及其他实战易错点，我们需要做如下补充设计：

### 1. 罗马音 (Romaji) 输入的终极解决方案
- **前端实时转换库 (Auto-Kana)**：在我们的 Web 前端中，引入非常轻量级的 JS 库（如 **WanaKana.js**）。它的作用是：当您在输入框里敲击英文键盘（如 `w-a-t-a-s-h-i`）时，**屏幕上的输入框里会实时自动变成假名 `わたし`**。
  - **优势**：您不需要去配置 Windows 操作系统的微软日语输入法，直接用平时打字习惯盲打拼音/罗马音即可。这也强迫您在屏幕上习惯看假名而不是英文字母。
- **AI 兜底批改 (Fallback)**：如果您就想以纯罗马音提交答案，我们会在给 Gemini 的 Prompt 中明确规定：“**如果用户答案是字母拼写的罗马音，请将其视同日语假名进行同标准批改，并在解析里强调它的正确假名写法。**”

### 2. 其他极易遗漏的核心设计细节
- **标点符号宽容度 (Punctuation Tolerance)**：
  - 日语的句号是 `。`，初学者很容易忘写或者写成中文的 `。` 英文的 `.`。
  - **建议**：UI 层面我们在输入框末尾默认缀上句号；或者在发送给 API 前，用 JS 正则表达式自动清理一遍末尾的标点和多余空格，不让您因为“标点错误”被扣分。
- **汉字与假名的双轨判定 (Kanji Validation)**：
  - 第 5 课学的单词可能只要求写假名，但如果 AI 批改时强求复杂的汉字，会打击自信心。
  - **建议**：在 Prompt 中规定：“只要用户的假名拼读是正确的，就必须给满分，但可以在讲解部分顺带提醒『写成汉字是XX会更好』”。
- **自然语序的兼容 (Flexible Answers)**：
  - 造句往往不是唯一答案（如 `わたしは あした 京都へ...` vs `あした わたしは 京都へ...` 或者 `に` 和 `へ` 的互换）。
  - Gemini 的最大优势就在于此，它不会像死板的代码那样只能认同标准答案。我们只需在提示词里加上“保持语序和近义助词的最大宽容度”。

## 四、 方案优势总结与建议

基于您提出的“AI 出题容易没上下文”和“零散访问效率低下”的痛点，这个设计完美解决了问题：

1. **上下文由本地注入 (Context Injection)**：AI 确实没有记忆，但我们有一个强大的本地 `data.json`。每次发请求，我们都把 `current_lesson` 和部分 `mistakes_book` 里的标签**拼接进 Prompt** 里发给 AI。AI 就“有了记忆”。
2. **批量化 (Batching)**：
   - **出题**：一次请求，生成本课的 10 道题缓存到本地内存。用户一道道做，不用卡顿。
   - **批改**：填空题本地鉴权，只有主观造句题才攒在一起（比如 3 道题）发给 AI 一次性批改。不仅极大地节省了您的 Gemini API Quota（虽然目前免费，但有 QPM 限制），而且体验更流畅。
3. **结构化输出 (JSON Forced)**：明确要求 AI 吐出 JSON，避免它说一堆废话（比如“好的，我现在给你出题...”），方便前端代码直接渲染 UI。

## 五、 极限优化 API 调用与上下文稳固方案 (Token Saving & Context Safety)

结合您对“上下文异常”和“Token 消耗”的顾虑，大模型最大的毛病就是**“幻觉”**（乱造词、超纲出题）和**“遗忘”**（不知道您学到哪了）。为了彻底解决这两个问题并最大化节省 API 请求（API 哪怕免费也有调用频率限制），我们的最终策略如下：

### 1. 彻底解决“上下文丢失”和“超纲出题 (幻觉)”的方案
每次调用 Gemini 时，我们发过去的 Prompt 结构必须像钢板一样严密。这套称为 **"Context Payload" (上下文预载架构)**：
- **零依赖会话 (Zero-Session State)**：我们**绝对不**使用 AI 平台自带的“连续对话上下文记忆功能”（也就是一问一答的 Chat History）。因为一旦网络断开或开启新一轮练习，历史消息就清空了。我们每次请求，都是把本地 `data.json` 里最核心的数据（学到几课拉、最近错了啥）作为**新的系统指令 (System Instruction)** 一次性发过去。只要本地数据在，AI 永远不会忘记您练到哪了。
- **静态教学大纲防超纲注入**：我们在本地（APP 代码里）准备一个非常精简的《大家的日语》重点目录结构表。当您学到第 14 课，我们在发给 AI 的 Prompt 里**强行拼接**一段："【白名单规则】：你只能使用 1-14 课的语法（以 `て形` 为止，坚决不准使用后面的字典形、假定形，不准使用复杂的 N3 词汇）。" 

### 2. 节省 API 访问频率与 Token 消耗的终极方案 (Caching)
- **本地防抖与题库预缓存 (Pre-fetch & Caching)**：
  - 这点极其关键！我们不需要每次您点击“下一题”都去请求一次 AI。
  - **初次进入页面**：触发一次批量拉取，比如要求 AI “立刻生成 10 道第 6 课的造句题”。收到的这 10 道题会被**缓存在您电脑浏览器的内存里**。
  - **做题过程中**：您做这 10 道题时，完全是零网络请求的、秒开的（像单机游戏一样）。
  - **批量批改再触发**：只有当您把这 10 道题做完，或者点击“提交这批答案去批改”时，才发生第二次网络请求。将 10 道题的批改一次性打包发过去。
- **混合出题模型 (Hybrid Generation)**：
  - 对于最简单的 Level 1 (助词选择填空题)，我们可以干脆用爬虫或者开源题库，在本地写死一套 50 课的静态题库。这类题完全不用消耗 Gemini API，所有 API 额度全留给“造句批改”和“情景对话”这种真正需要 AI 算力的场景。

### 3. 一些日语学习的进阶建议 (Bonus Features)
结合您赴日工作的目标，APP 还可以加入以下进阶能力：
- **五十音魔鬼特训前置关卡**：这是不能遗漏的基石。在 Level 1 应当有个前置关卡——随机平/片假名快问快答（纯本地实现，零 API 消耗）。
- **口语影子跟读 (Shadowing)**：浏览器内置了极其强大的 `Web Speech API`。不需要额外花钱接大厂语音，当您造对了一句日语，App 会用标准的日音把句子朗读出来，您看着屏幕跟着读，这种听说联动比纯打字在职场有用 100 倍。

## 六、 您的 API Key 处理

您申请的秘钥可以直接用在我们的架构中。代码只跑在您自己的电脑上，API 请求全部从您的本地浏览器发出（CORS），所以把 API Key 写在本地的一个配置文件中（并用 Git `.gitignore` 保护起来）是**绝对安全**的。

# Phase 2: Vue 3 架构重构与深度大纲引擎 (核心升级计划)

基于第一期开发后的系统体验反馈，系统将彻底由 Vanilla JS 向 Vue 3 核心重构，并引入深度的业务功能升级：

## 1. Vue 3 架构大重构
为了长期的可维护性和拓展性，系统将弃用原有的原生 JavaScript 操作 DOM 方式。
**重构方案**：
*   **工程化脚手架**：采用 `Vite` 构建现代化的 Vue 3 工程。
*   **状态管理闭环**：利用 Vue 的响应式 API (`ref`, `reactive`) 或轻量级全局状态（Composable 模式）替代庞大杂乱的 `LocalState` 类，实现数据驱动视图。
*   **组件化拆分**：
    *   `Dashboard.vue`：进度总览、选课、设置难度。
    *   `SyllabusManager.vue`：课程大纲、知识点、题型的 CRUD 管理面板。
    *   `TrainingEngine.vue`：答题流转控制中枢。
    *   `QuestionCard.vue` (多态)：分化为填空卡片、造句卡片等独立组件。

## 2. 深度《大家的日语》内置大纲字典 (The Deep Syllabus)
**问题**：直接扔给 AI 去出题，AI 会脱离书本原有框架，出一些虽在语法内、但超出该课词汇量的生词，或者忽略日语特有的“语感”、“职场敬语”等隐性重点。
**方案**：在项目中构建并持久化维护一份深度的 `src/data/syllabus.json`。
**数据骨架设计**：
```json
{
  "question_types": [
    { "id": "q_fill", "name": "基础粒子填空", "desc": "考察助词逻辑" },
    { "id": "q_translate", "name": "降维造句", "desc": "强制附带单词提示卡的翻译" },
    { "id": "q_conversation", "name": "职场情景对话", "desc": "A与B的工作沟通补全" }
  ],
  "lessons": [
    {
      "id": 1,
      "title": "第 1 课",
      "grammar_points": ["名词1 は 名词2 です", "名词1 は 名词2 じゃありません", "名词 も"],
      "hidden_knowledge": [
        "发音注意：助词的「は」要读作「wa」",
        "职场礼仪：初次见面寒暄时多用「初めまして」而非随意打招呼"
      ],
      "enabled_question_types": ["q_fill", "q_translate"]
    }
  ]
}
```
**出题引擎结合**：当您选择第 1 课的 `q_conversation` 题型时，AI 的 Prompt 将被强制注入该课的 `grammar_points` 和 `hidden_knowledge`。这让大模型不仅能考察显性语法，还能“顺手”考查发音易错点和职场常识。

## 3. 按题型的分段流转与颗粒度进度记录
**问题**：一套题目混杂输出，单课的答题状态缺乏颗粒度跟踪。
**方案**：
*   **分步流转**：彻底改变。进入某课后，下拉列表中展示该课支持的题型。点击“开始【填空题】挑战”，完成后即存储“第 X 课-填空题-已完成”。
*   **记忆细分**：如果切走复习他课，再切回来，您可以明确看到自己当前课哪些题型没做。

## 4. 自定义难度、加练与提示词注入
**方案**：
*   在触发生成的面板提供 **【难度等级】** 和 **【自定义指令】** 输入框。
*   在批改报告指出您某项 `grammar_point` 错误时，提供一个 **【专项加练】** 按钮，一键让原生模块针对这一个点再出 3 道题强化！

# Phase 3: 架构深化、错题本全翻新与 Element Plus 引擎 (已完成)

为了解决路由跳转阻断生成、组件管理复用差以及界面交互的现代化诉求，我们目前已全面完成了功能纵深扩展：

## 1. 跨路由页面状态保持 (Stateful Requests)
**问题**：当前在 `TrainingEngine.vue` 组件内部触发 AI 请求，如果用户切换了 Router (比如去大纲页面看了一眼)，该组件会被 Destroy 销毁，导致昂贵的出题请求被切断，题目丢失。
**重构方案**：
*   在 Pinia 中创建 `useTrainingStore`。
*   将出题状态（`isLoading`, `exercises[]`, `userAnswers{}`, `currentPhase`）全盘上移到全局容器中。
*   组件只负责 Render。使得您发起出题后，就算切换到别的页面，生成流仍然在后台异步执行；切换回来时，屏幕依然完美停留在那组题的答题画面或最后一道题的锚点。

## 2. Element Plus 工业级重塑
**方案**：
既然要搭建丰富的题型大纲 CRUD 图谱和网格导航，使用第三方 UI 组件库 `Element Plus` 是最优解。
*   利用 `el-menu` 搭建稳固侧边栏或顶栏体系。
*   利用 `el-dialog` 和 `el-form` 重写厚重的《大纲管理模块》，对每个课时的语法和题型集合进行可视化配置。

## 3. 面向题目的网格导航与题量控制
**方案**：
*   **题控台扩充**：Dashboard 的出题面板新增**【生成题目数量】**，可以自定义要出 5 题还是 20 题。
*   **答题流转网格**：弃用简单的 Prev/Next 按钮。在答题屏幕右侧增加一个类似驾考宝典的【答题格子矩阵 (Grid Nav)】（例如 1~15号方块）。如果用户想跳过第 2 题先做最后一道题，只需点击格子直达。

## 4. 动态题型集 CRUD
**方案**：
*   建立一个 `question_types_pool.json`。
*   允许用户在界面上动态创造新题型：例如自己创造了一个新的【日本文化理解选择题】，写入题型池。
*   然后在 Lesson 1 中，将这个题型拉入其可用的出题子集内。AI Engine 最终根据这个数据动态组合出题。

## 5. Vocab-Hints (单词提示卡) 屏蔽层机制
**诉求**：学习的核心是回忆而非照抄提示。
**方案**：已实现前端 CSS 悬浮毛玻璃 (Blur) 遮罩，默认隐藏生词读音与日文汉字，强迫训练记忆提取。

# Phase 4: 私人定制视听体验与 ACG 文化聚合 (当前启动)

为了满足用户在职场与私人环境下的心情调节，以及更深度的二次元文化沉浸，在此扩展此阶段：

## 1. ACG 暗黑风主题与摸鱼伪装模式 (UI/UX)
**问题**：用户希望在公司学习时不引人瞩目（普通暗黑模式），而在家能有心情愉悦的二次元暗黑风格。
**方案**：
*   **CSS 变量驱动**：提取全站的核心色调为 CSS Variable (`--el-bg-color`, `--el-color-primary` 等)。
*   **双模式一键切换**：设计两种套件：
    *   **ACG 模式**：柔和暗灰蓝背景，带有微透明毛玻璃特效，主色调可能使用Galgame常见的霓虹粉或幽兰色。
    *   **Work 模式**：深沉、极简的 VSCode 极客感无色彩黑白灰。
*   **快捷键介入**：注册前端 `keydown` 监听器，提供类似 `Alt + D` 或 `Ctrl + \` 的老板键级一键切换，并支持右下角悬浮按钮。修复原来未对齐的红色圆章删除按钮等边角 UI。

## 2. 二次元名台词双语每日一贴 (API & Content)
**问题**：寻找一个能够稳定返回【日文台词】+【中文翻译】+【出处】的免费 API。
**方案**：由于传统的免费名言接口（如 Hitokoto）多半仅返回中文且二次元内容日渐庞杂，我们将继续榨干**现成免配的 Google Gemini API** 价值：
*   新增一个 `GenerateQuoteSkill.js`。每次页面加载或点击“换一句”时，命令大模型：“随机抽取一句经典番剧或知名 Galgame 中的感人/热血/治愈名言。返回 JSON 包含 `japanese`, `chinese`, `source`, `character`”。
*   在 `Dashboard.vue` 顶部开辟一块优雅的卡片区域进行渲染，附带刷新按钮。

## 3. Web Speech 发音接口的全局下放
**问题**：原生 TTS 功能此前仅局限在批改报告出现，前台感知不强。
**方案**：
*   将原有的 `window.speechSynthesis` 封装为全局的组合式 API（Composable）或者通过 Pinia Store 提供全局调用。
*   为 `Dashboard` 上新出的名言栏，以及做题途中的核心区域（如果可以的话）附带播放原生语音的按钮，提升耳朵的磨练频次。

## 4. 艾宾浩斯智能复习算法 (SRS) & 语音录入 (Future)
*   预留方向：继续完善记忆曲线与麦克风实机判卷（通过 Web SpeechRecognition API）。
